## 1. 배경 및 목적

### 1.1 현재 문제

수탁업체 실무자의 실제 업무 흐름을 분석한 결과, 현재 설계에 근본적 문제가 발견됨:


| #   | 문제                                   | 상세                                                                                               |
| --- | ------------------------------------ | ------------------------------------------------------------------------------------------------ |
| 1   | **영상 편집과 콘텐츠 작성의 인위적 분리**            | "위탁기관에서 받은 영상으로 인스타 릴스 만들어서 올려야 해" — 자막, 규격 변환, 제목 작성, 게시가 하나의 연속 작업인데, 현재는 편집/미디어/게시가 분리된 설계    |
| 2   | **플랫폼별 미디어 분리 불가**                   | 1 Content에 공용 `media_urls` 배열 → YouTube(16:9 영상), Instagram(9:16 릴스), X(이미지) 등 플랫폼별 다른 미디어 사용 불가 |
| 3   | **ContentCreatePage/EditPage 코드 중복** | 1,246줄 + 993줄 ≒ 2,239줄 중 약 70%가 동일 로직                                                            |
| 4   | **미디어 라이브러리의 불필요한 업로드 기능**           | 업로드는 콘텐츠 작성 시에만 발생하는데, 미디어 라이브러리에 별도 업로드 경로가 있어 실무 동선과 불일치                                       |
| 5   | **MinIO 직접 의존**                      | 개발 환경에서 Docker 컨테이너 필요, 업로드 디버깅 복잡, 프로덕션에서는 S3가 현실적                                              |
| 6   | **승인 권한 구조의 경직성**                    | CD(위탁기관 담당자)가 유일한 승인권자로 설계되어 있으나, 일상적인 콘텐츠까지 위탁기관 승인을 받으면 게시 속도 저하. 기관별로 승인 구조를 유연하게 설정할 수 없음    |


### 1.2 협의 결정사항

사용자와의 검토를 통해 다음 사항이 합의됨:

1. **ContentCreatePage + ContentEditPage → ContentEditorPage 통합**: 새 작성/수정이 같은 화면, URL 파라미터로 구분
2. **미디어 라이브러리 → 조회·재사용·추적 전용**: 업로드 버튼 제거, 콘텐츠 작성 시 업로드한 파일이 자동으로 라이브러리에 저장
3. **MinIO → 로컬 파일시스템 기본 + S3 전환 가능한 추상화**: 개발 시 Docker 불필요
4. **content_variants 테이블 도입**: 플랫폼별 파생본 관리 (제목, 본문, 미디어, 자막, 썸네일 각각 분리)
5. **승인 권한 유연화**: AM(수탁업체 관리자)을 기본 승인권자로 설정. 기관별 설정에서 "CD 사전 검수 필요" 토글(ON/OFF)을 제공하여, 위탁기관이 사전 검수를 원하는 경우 CD → AM 2단계 승인 가능. CD는 기본적으로 대시보드/리포트 조회 + 옵션 검수 역할
6. **variant별 반려 피드백**: 승인 반려 시 특정 variant에 대한 코멘트를 남길 수 있도록 approval_history에 variant_id(nullable) 추가. AM/CD가 "Instagram 버전 자막 수정 필요" 같은 variant 단위 피드백 가능

### 1.3 목표 업무 흐름

```
위탁기관 → 원본 소재 전달 → 수탁업체 실무자(AO)

[ContentEditorPage — 단일 화면 레이아웃]

┌─────────────────────────────────────────────────────┐
│ 상단: 공통 정보                                       │
│   제목(공통), 본문(공통), 해시태그, 소스 미디어 업로드    │
│   [전체 동일 게시 □ / 플랫폼별 커스터마이즈 ■] 토글     │
├─────────────────────────────────────────────────────┤
│ 하단: 플랫폼 탭 (토글이 "커스터마이즈"일 때 활성)        │
│   [YouTube] [Instagram] [Facebook] [X] [네이버]       │
│   각 탭: 채널 선택, 제목·본문 오버라이드                 │
│          플랫폼별 미디어, AI 도구, 미리보기              │
├─────────────────────────────────────────────────────┤
│ 푸터: [임시 저장]  [검수 요청] ← 확인 모달              │
└─────────────────────────────────────────────────────┘

  "전체 동일 게시" 모드:
    - 공통 제목/본문/미디어로 선택한 채널에 동일 게시
    - variant 편집 불필요 (빠른 게시 시나리오)
    - 내부적으로 채널별 variant 자동 생성

  "플랫폼별 커스터마이즈" 모드:
    - 플랫폼 탭에서 각 variant별 제목/본문/미디어 편집
    - AI 도구: 제목 생성, 톤 변환, 자막, 해시태그 등

  비선형 편집 가능:
    - 공통 정보 ↔ 플랫폼 탭 자유 전환
    - 한 탭에서 다른 탭으로 바로 이동

[승인 워크플로우]
  기본: AO → AM(수탁업체 관리자) 승인 → 게시
  옵션: CD 사전 검수 ON 시 → AO → CD 검수 → AM 승인 → 게시
  반려 시: variant별 코멘트 가능 ("Instagram 자막 수정 필요")

[게시]
  AO → 게시 버튼 → variant별 플랫폼 어댑터 호출 → 각 채널에 게시
  → 게시 이력 탭에서 채널별 결과 확인
```

### 1.4 레거시 데이터 호환

variant 도입 이전에 생성된 기존 콘텐츠에 대한 처리:

- **데이터 마이그레이션**: 기존 `platforms`/`channel_ids` → variant 레코드 자동 변환 (제목/본문 동일)
- **에디터 진입 시**: 마이그레이션된 콘텐츠는 "기존 콘텐츠 — 플랫폼별 편집이 가능합니다" 안내 배너 표시
- **목록/상세**: variants 빈 배열이면 기존 필드에서 역산하여 표시 (fallback)

## 3. 목표 데이터 모델

### 3.1 변경 후 구조

```
contents (프로젝트)
  ├─ title, body (공통)
  ├─ source_media_id (원본 소재)
  ├─ status (DRAFT → APPROVED → PUBLISHED)
  │
  └─ content_variants[] (1:N, 플랫폼별 파생본)

       │
       ├─ variant_media[] (1:N, 역할별 미디어)
       │    ├─ media_asset_id + role (SOURCE/EDITED/SUBTITLE/THUMBNAIL/EFFECT)

       └─ publish_results[] (1:N, 게시 결과)
```

### Sprint 2: 백엔드 API/서비스 재설계

**목적:** variant 기반 게시 플로우 구현, 하위호환 유지.

#### 신규 API

```
POST   /contents/{id}/variants              — variant 추가
GET    /contents/{id}/variants              — variant 목록
PUT    /contents/{id}/variants/{vid}        — variant 수정
DELETE /contents/{id}/variants/{vid}        — variant 삭제
POST   /contents/{id}/variants/{vid}/media  — variant에 미디어 연결
DELETE /contents/{id}/variants/{vid}/media/{mid} — 미디어 해제
```

#### 기존 API 하위호환

- `POST /contents` body에 `variants[]` 옵션 추가 (없으면 기존 필드에서 자동 생성)
- `GET /contents/{id}` response에 `variants[]` 포함
- publish 시: variants 있으면 variant 기반, 없으면 기존 fallback

#### "전체 동일 게시" 모드 지원

- `POST /contents` body에 `uniform_publish: true` + `channel_ids[]`를 보내면, 서버에서 채널별 variant를 자동 생성 (제목/본문/미디어 동일)
- 프론트엔드 "전체 동일 게시" 토글 ON 시 이 모드 사용

#### 승인 권한 변경

- `POST /approvals/{id}/approve`, `POST /approvals/{id}/reject`: 기본 `require_roles(AM)`, 기관 설정에서 CD 검수 ON 시 CD도 허용
- 기관별 설정: `organization_settings`에 `require_cd_review: bool` (기본값 false) 추가
- `require_cd_review=true`일 때 승인 흐름: AO → CD 검수 → AM 최종 승인

## 5. 마이그레이션 전략

### 원칙

- **비파괴적 변경 우선**: 기존 컬럼 즉시 삭제하지 않음
- **Phase A/B 분리**: 테이블 추가(A)와 데이터 이동(B)을 별도 마이그레이션으로
- **하위호환 유지**: 새 코드는 variants 사용, 기존 API도 자동 변환으로 동작
- **롤백 가능**: 각 마이그레이션에 downgrade 함수 구현

### 호환 모드 동작

```python
# 서비스 레이어: variants가 있으면 variant 기반, 없으면 기존 로직
if content.variants:
    # 새 경로: variant별 게시
    for variant in content.variants:
        title = variant.title or content.title     # 오버라이드 없으면 공통 사용
        body = variant.body or content.body
        adapter.publish(title, body, variant.media)
else:
    # 기존 경로: content 레벨 게시 (fallback, 레거시 호환)
    for channel_id in content.channel_ids:
        adapter.publish(content.title, content.body, content.media_urls)

# "전체 동일 게시" 모드 (uniform_publish=true):
# → 콘텐츠 생성 시 channel_ids에서 채널별 variant 자동 생성
# → 이후 일반 variant 게시 경로와 동일하게 처리

# 승인 권한 분기:
# org_settings.require_cd_review == True:
#   AO → PENDING_REVIEW → CD 검수(IN_REVIEW) → AM 승인(APPROVED)
# org_settings.require_cd_review == False (기본):
#   AO → PENDING_REVIEW → AM 승인(APPROVED)
```

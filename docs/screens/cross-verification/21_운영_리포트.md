## 5. 쿼리 로직 검증

### 5.1 list_reports (Repository)
- `org_id` 필터 ✅
- `period`, `status` 선택적 WHERE 조건 ✅
- count 쿼리와 data 쿼리 필터 일치 ✅
- `created_at DESC` 정렬 ✅
- offset/limit 페이지네이션 ✅

### 5.2 get_by_id (Repository)
- `report_id + org_id` 이중 필터로 크로스 테넌트 접근 차단 ✅

### 5.3 get_analytics_data (Repository — Celery 태스크 내 사용)
- `org_id` + `PublishResultStatus.SUCCESS` + 날짜 범위 필터 ✅
- Channel JOIN으로 platform 그룹핑 ✅
- `func.date()` 사용하여 date 비교 (timestamp → date 캐스팅) ✅

### 5.4 get_sentiment_summary (Repository)
- `org_id` + 날짜 범위 필터 ✅
- `Comment.sentiment` 그룹핑 ✅

### 5.5 get_top_contents (Repository)
- engagement = views + likes + shares + comments_count 합산 ✅
- Content-PublishResult JOIN ✅
- `SUCCESS` 상태 필터 ✅
- 상위 5건 제한 (engagement DESC) ✅

### 5.6 Celery Task (generate_report_task)
- `sync_session_factory` 사용 (동기 세션, Celery 워커 컨텍스트) ✅
- Repository와 동일한 쿼리 패턴 복제 (platform, sentiment, top contents) ✅
- AI 응답 파싱 실패 시 fallback 텍스트 제공 (Human-in-the-Loop) ✅
- job 실패 시 report.status=DRAFT로 설정 (수동 편집 가능) ✅
- PDF 생성 실패 시 리포트 자체는 성공 처리 (pdf_url만 null) ✅
- try/finally로 session.close() 보장 ✅
- 지수 백오프 재시도 (max_retries=1, delay=60s) ✅

### 5.7 비즈니스 로직 검증 (Service)
- `update_report`: FINALIZED 상태 수정 불가 ✅
- `finalize_report`: 이미 FINALIZED면 None 반환 ✅
- `delete_report`: FINALIZED 삭제 불가 ✅
- `get_pdf_url`: org_id 검증 포함 ✅

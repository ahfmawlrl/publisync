## 발견 및 수정 사항

### 이슈 1: hashtags 필드 FE→BE 불일치 (수정됨)

| 항목 | 수정 전 | 수정 후 |
|------|---------|---------|
| BE `ContentCreateRequest` | `hashtags` 필드 없음 → silently dropped | `hashtags: list[str] \| None = None` 추가 |
| BE `ContentUpdateRequest` | `hashtags` 필드 없음 | `hashtags: list[str] \| None = None` 추가 |
| BE `ContentResponse` | `hashtags` 필드 없음 | `hashtags: list[str] = []` 추가 |
| BE `content_service.create_content` | hashtags 무시 | `metadata_["hashtags"]`로 저장 |
| BE `content_service.update_content` | hashtags 무시 | `metadata_["hashtags"]`로 저장 |
| BE `_to_content_response` | hashtags 미반환 | `meta.get("hashtags", [])` 반환 |
| FE `ContentRecord` | `hashtags` 필드 없음 | `hashtags: string[]` 추가 |

**저장 방식:** Content 모델에 전용 컬럼 없으므로 `metadata_` JSONB에 `{"hashtags": [...]}` 형태로 저장 (마이그레이션 불필요).

### 이슈 2: 임시 저장 후 이동 대상 (설계서 수정)

| 항목 | 수정 전 | 수정 후 |
|------|---------|---------|
| 설계서 §3.3 | "콘텐츠 목록으로 이동" | "콘텐츠 상세 페이지로 이동" |
| 설계서 §5.6 | "콘텐츠 목록 이동" | "콘텐츠 상세 이동" |

FE `navigate(`/contents/${result.id}`)` → 생성 직후 상세 페이지로 이동하는 것이 UX 상 더 적절.

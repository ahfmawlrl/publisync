# 워크플로우 설정 (WorkflowSettingsPage)

> **경로:** `/approvals/settings`
> **소스:** `frontend/src/features/approvals/pages/WorkflowSettingsPage.tsx`
> **접근 권한:** SA (시스템 관리자), AM (수탁업체 관리자), CD (위탁기관 담당자)
> **Phase:** 1-A (최소 MVP)
> **관련 기능:** F09 승인워크플로우

---

## 1. 화면 개요

수탁업체 관리자(AM) 또는 시스템 관리자(SA)가 콘텐츠 승인 워크플로우를 구성하는 화면이다.
현재 Phase 1-A에서는 기본 1단계 승인(CD 승인) 워크플로우의 이름과 활성화 상태를 관리할 수 있으며,
워크플로우 단계 상세 설정(최대 3단계 검수, Fast Track, 자동 반려 조건)은 Phase 1-B에서 지원 예정이다.

목표 워크플로우 파이프라인:
**작성 -> 검토요청 -> 검수 -> 승인 -> 게시**

### 핵심 원칙
- **Phase 점진 투입:** 현재는 기본 1단계 승인만 지원, 고급 기능은 Phase 1-B에서 확장
- **Fallback 필수:** 워크플로우 설정 실패 시에도 기본 1단계 승인으로 운영 가능
- **멀티테넌트 격리:** 워크플로우는 기관(organization)별로 독립 관리

## 6. Phase 1-B 확장 예정 기능

현재 Phase 1-A에서는 구현되지 않았으나, 설계 목표로 계획된 기능 목록이다.

### 6.1 비주얼 파이프라인 빌더

| 항목 | 내용 |
|------|------|
| **Phase** | 1-B |
| **목표** | 승인 단계를 시각적으로 구성하는 드래그 앤 드롭 파이프라인 |
| **파이프라인 단계** | 작성 -> 검토요청 -> 검수 -> 승인 -> 게시 |
| **표시 형태** | 수평 스텝 바 + 각 단계 설정 패널 |

### 6.2 검수 단계 설정 (최대 3단계)

| 항목 | 내용 |
|------|------|
| **Phase** | 1-B |
| **목표** | 최대 3단계까지 검수 단계를 추가 구성 |
| **단계별 설정** | 검수자 역할 지정, 병렬/순차 승인, 자동 에스컬레이션 |

#### 검수 단계 구성 예시

| 단계 | 검수자 | 조건 | 설명 |
|------|--------|------|------|
| 1단계 | AO (수탁업체 실무자) | 내용 검토 | 기본 내용 확인 |
| 2단계 | AM (수탁업체 관리자) | 품질 검수 | 품질 기준 충족 확인 |
| 3단계 | CD (위탁기관 담당자) | 최종 승인 | 게시 최종 승인 |

### 6.3 Fast Track 설정 (긴급 콘텐츠)

| 항목 | 내용 |
|------|------|
| **Phase** | 1-B |
| **목표** | 긴급 콘텐츠(`is_urgent=true`)의 승인 단계 간소화 |
| **설정 옵션** | 단계 스킵, 자동 승인 조건, 긴급 알림 채널 지정 |

### 6.4 자동 반려 조건

| 항목 | 내용 |
|------|------|
| **Phase** | 1-B |
| **목표** | 특정 조건 충족 시 자동으로 반려 처리 |
| **조건 예시** | 검수 기한 초과, AI 검수 심각 이슈 발견, 필수 항목 누락 |

## 8. 연결 흐름 (Connected Flows)

### 8.1 워크플로우 설정 → 승인 프로세스 적용 흐름

```
[워크플로우 설정] (/approvals/settings)
       │
       ├──(워크플로우 생성/수정)──→ PUT /approvals/workflows
       │                              │
       │                              └──→ [워크플로우 목록 갱신]
       │
       └──(활성 워크플로우)──→ [콘텐츠 검토 요청 시 자동 적용]
                                    │
                                    └──→ [승인 대기 목록] (/approvals)
```

### 8.2 워크플로우와 콘텐츠 승인 연계 흐름

```
[워크플로우 활성화]
       │
       ↓
[AO: 콘텐츠 검토 요청] ──→ 활성 워크플로우 기반 승인 프로세스 시작
       │
       ↓
[승인 단계 1] ──→ (Phase 1-A: CD 1단계 승인만 적용)
       │
       ├──(승인)──→ [자동 게시]
       │
       └──(반려)──→ [AO/AM: 수정 후 재요청]
```

### 8.3 설정 메뉴 내 위치

```
[설정 메뉴]
       │
       ├──→ [기관 설정] (/settings/organization)
       │
       ├──→ [워크플로우 설정] (/approvals/settings)  ← 현재 화면
       │
       ├──→ [알림 설정] (/settings/notifications) [Phase 1-B]
       │
       └──→ [사용자 관리] (/settings/users)
```

## 10. 데이터 타입

### 10.1 워크플로우 레코드 (WorkflowRecord)

```typescript
interface WorkflowRecord {
  id: string;                              // 워크플로우 ID (UUID)
  organization_id: string;                 // 소속 기관 ID
  name: string;                            // 워크플로우 이름
  steps: Record<string, unknown>[] | null; // 승인 단계 배열 (Phase 1-B에서 상세 구현)
  is_active: boolean;                      // 활성화 여부
  created_at: string;                      // 생성 일시 (ISO 8601)
}
```

### 10.2 워크플로우 수정 요청

```typescript
interface WorkflowUpdateData {
  name?: string;                           // 워크플로우 이름
  steps?: Record<string, unknown>[];       // 승인 단계 배열
  is_active?: boolean;                     // 활성화 여부
}
```

### 10.3 워크플로우 단계 구조 (Phase 1-B 목표)

```typescript
// Phase 1-B에서 steps 배열의 각 요소 구조
interface WorkflowStep {
  step: number;           // 단계 번호 (1, 2, 3)
  role: string;           // 검수자 역할 (AO, AM, CD)
  action: string;         // 액션 유형 (review, approve)
  is_required: boolean;   // 필수 여부
  timeout_hours?: number; // 자동 에스컬레이션 시간 (시간)
  auto_approve?: boolean; // Fast Track 자동 승인 여부
}
```

## 12. 오류 처리 및 Fallback

| 상황 | 처리 |
|------|------|
| **워크플로우 목록 로딩 실패** | Card loading 스켈레톤 → 데이터 없으면 `Empty` 표시 |
| **워크플로우가 없는 경우** | `Empty` (description="설정된 워크플로우가 없습니다") |
| **워크플로우 저장 성공** | `message.success('워크플로우가 저장되었습니다')` toast |
| **워크플로우 저장 실패** | `message.error('워크플로우 저장에 실패했습니다')` toast |
| **폼 유효성 검사 실패** | "이름을 입력하세요" 인라인 에러 메시지 (Ant Design Form 기본) |
| **네트워크 오류** | TanStack Query 기본 재시도 (3회) |
| **Phase 1-B 기능 접근 시도** | Phase 안내 메시지로 미지원 기능 안내 |

## 13. 사용 시나리오

### 시나리오 1: AM이 기본 워크플로우 생성

1. AM이 사이드바 > 설정 > 워크플로우 설정 클릭 → `/approvals/settings` 진입
2. 좌측 "현재 워크플로우" 카드에서 기존 워크플로우 확인 (없으면 Empty 표시)
3. 우측 "워크플로우 편집" 카드에서 이름 입력: "기본 승인 워크플로우"
4. 활성화 스위치 ON (기본값)
5. [저장] 클릭 → "워크플로우가 저장되었습니다" 메시지
6. 좌측 목록에 새 워크플로우 자동 반영

### 시나리오 2: AM이 워크플로우 비활성화

1. 좌측 목록에서 기존 워크플로우 확인 (활성: 예)
2. 우측 편집 폼에서 해당 워크플로우 이름 입력
3. 활성화 스위치 OFF
4. [저장] 클릭 → 워크플로우 비활성화됨
5. 이후 콘텐츠 검토 요청 시 해당 워크플로우 미적용

### 시나리오 3: CD가 워크플로우 현황 확인

1. CD가 워크플로우 설정 페이지 진입
2. 좌측 "현재 워크플로우" 카드에서 활성 워크플로우 목록과 단계 수 확인
3. 자신이 검수자로 지정된 워크플로우가 활성화되어 있는지 확인
4. 편집은 AM/SA만 가능하므로, 변경 필요 시 AM에게 요청

### 시나리오 4: SA가 여러 기관의 워크플로우 관리

1. SA가 워크스페이스 전환 (상단바에서 기관 선택)
2. 각 기관별 워크플로우 설정 페이지 진입
3. 기관별 승인 워크플로우를 개별 구성
4. 기관 A: 기본 1단계, 기관 B: 비활성화 등 기관별 독립 설정

# 콘텐츠 수정 (ContentEditPage)

> **경로:** `/contents/:id/edit`
> **소스:** `frontend/src/features/contents/pages/ContentEditPage.tsx`
> **접근 권한:** AM (수탁업체 관리자), AO (수탁업체 실무자)
> **Phase:** 1-A (최소 MVP)
> **관련 기능:** F01 통합게시, F09 승인워크플로우

---

## 1. 화면 개요

기존 콘텐츠를 수정하는 화면이다. 새 콘텐츠 작성(ContentCreatePage)과 동일한 폼 구조를
사용하되, 기존 콘텐츠 데이터를 미리 채워(pre-fill) 표시한다.
`DRAFT`(초안) 또는 `REJECTED`(반려) 상태의 콘텐츠만 수정 가능하며,
수정 완료 후 상세 페이지로 자동 이동한다.

### 핵심 원칙
- **기존 데이터 보존:** 수정 화면 진입 시 서버에서 최신 데이터를 조회하여 폼에 반영
- **상태 기반 접근:** DRAFT/REJECTED 상태만 수정 가능, 그 외 상태는 접근 차단
- **수정 후 확인:** 저장 성공 시 상세 페이지로 이동하여 변경 내용 즉시 확인

| 항목 | 내용 |

## 5. 폼 제출 로직 (handleSubmit)

### 5.1 처리 흐름

```
[저장 버튼 클릭] ──→ Form.onFinish 호출
       │
       ↓
  id 유효성 확인 (없으면 중단)
       │
       ↓
  updateMutation.mutateAsync({
    id,
    data: {
      title,                          // 필수
      body,                           // 선택
      platforms: [] || [],             // 기본 빈 배열
      scheduled_at: ISO8601 || null    // dayjs → ISO 변환
    }
  })
       │
  ┌────┴────┐
  ↓         ↓
[성공]    [실패]
  │         │
  ↓         ↓
토스트     토스트
"콘텐츠가  "콘텐츠 수정에
수정됨"    실패했습니다"
  │
  ↓
navigate('/contents/${id}')
(상세 페이지로 이동)
```

### 5.2 데이터 변환

| 필드 | 변환 처리 |
|------|----------|
| `title` | `values.title as string` |
| `body` | `values.body as string \| undefined` |
| `platforms` | `(values.platforms as string[]) \|\| []` |
| `scheduled_at` | `values.scheduled_at ? .toISOString() : null` |

## 7. 연결 흐름 (Connected Flows)

### 7.1 수정 화면 진입/이탈 흐름

```
[콘텐츠 상세] ──(수정 버튼)──→ [콘텐츠 수정]
(/contents/:id)                (/contents/:id/edit)
(DRAFT/REJECTED만)                   │
                                ┌────┴─────┐
                                ↓          ↓
                           [저장 성공]  [취소 클릭]
                                │          │
                                ↓          ↓
                           [상세 페이지 복귀]
                           (/contents/:id)

[콘텐츠 목록] ──(수정 아이콘)──→ [콘텐츠 수정]
(/contents)                    (/contents/:id/edit)
(DRAFT/REJECTED만)
```

### 7.2 수정 → 재검토 요청 흐름

```
[반려된 콘텐츠 상세] ──(수정)──→ [콘텐츠 수정]
                                      │
                                      ↓
                                 [내용 수정 후 저장]
                                      │
                                      ↓
                                [콘텐츠 상세] (status: DRAFT)
                                      │
                                      ↓
                                 [검토 요청 버튼 활성화]
                                      │
                                      ↓
                                 [재검토 요청 가능]
```

### 7.3 새 콘텐츠 작성과의 비교

```
[새 콘텐츠 작성]                    [콘텐츠 수정]
ContentCreatePage                ContentEditPage
(/contents/create)               (/contents/:id/edit)
       │                                │
       ├── useCreateContent()           ├── useContent() (데이터 조회)
       ├── POST /contents              ├── useUpdateContent()
       ├── 빈 폼                        ├── PUT /contents/:id
       └── → 목록으로 이동               ├── 프리필된 폼
                                       └── → 상세로 이동
```

## 8. API 엔드포인트 연결

| 동작 | HTTP | 엔드포인트 | Hook | Phase |
|------|------|-----------|------|-------|
| 콘텐츠 상세 조회 | GET | `/contents/:id` | `useContent()` | 1-A |

### 8.1 상세 조회 API

| 항목 | 내용 |
|------|------|
| **응답 형태** | `ApiResponse<ContentRecord>` |
| **캐시 키** | `['contents', id]` |
| **활성 조건** | `enabled: !!contentId` |
| **용도** | 폼 프리필 데이터 조회 |

### 8.2 수정 API

| 항목 | 내용 |
|------|------|
| **요청 형태** | `{ id: string, data: ContentUpdateData }` |
| **응답 형태** | `ApiResponse<ContentRecord>` |

| **용도** | 콘텐츠 필드 업데이트 |

## 9. 데이터 타입

### 9.1 콘텐츠 수정 요청 (ContentUpdateData)

```typescript
interface ContentUpdateData {
  title?: string;                              // 제목 (선택적 업데이트)
  body?: string;                               // 본문/설명문
  platforms?: string[];                        // 게시 대상 플랫폼 배열
  channel_ids?: string[];                      // 채널 ID 배열
  scheduled_at?: string | null;                // 예약 게시 일시 (ISO 8601 또는 null)
  platform_contents?: Record<string, unknown>; // 플랫폼별 맞춤 콘텐츠
  media_urls?: string[];                       // 미디어 파일 URL 배열
  metadata?: Record<string, unknown>;          // 추가 메타데이터
}
```

### 9.2 콘텐츠 레코드 (ContentRecord)

```typescript
interface ContentRecord {
  id: string;                                  // 콘텐츠 고유 ID (UUID)
  organization_id: string;                     // 소속 기관 ID
  title: string;                               // 콘텐츠 제목
  body: string | null;                         // 본문/설명문
  status: string;                              // 콘텐츠 상태 (12가지)
  platforms: string[];                         // 게시 대상 플랫폼 배열
  channel_ids: string[];                       // 채널 ID 배열
  scheduled_at: string | null;                 // 예약 게시 일시 (ISO 8601)
  author_id: string;                           // 작성자 ID
  platform_contents: Record<string, unknown> | null;  // 플랫폼별 맞춤 콘텐츠
  metadata: Record<string, unknown> | null;    // 추가 메타데이터
  ai_generated: boolean;                       // AI 제안 사용 여부
  media_urls: string[];                        // 미디어 파일 URL 배열
  created_at: string;                          // 작성일 (ISO 8601)
  updated_at: string;                          // 수정일 (ISO 8601)
}
```

## 10. 상태 관리

| 영역 | 도구 | 설명 |
|------|------|------|
| **URL 파라미터** | `useParams()` | `:id` -- 수정 대상 콘텐츠 ID 추출 |
| **폼 상태** | Ant Design `Form.useForm()` | 모든 입력 필드 값 관리 |
| **콘텐츠 데이터** | TanStack Query `useContent()` | 기존 콘텐츠 데이터 조회 (프리필용) |
| **수정 뮤테이션** | TanStack Query `useUpdateContent()` | 콘텐츠 수정 mutation |
| **프리필 동기화** | `useEffect` | `content` 로드 시 `form.setFieldsValue()` 호출 |
| **네비게이션** | `useNavigate()` | 상세 페이지 이동 |
| **메시지** | `App.useApp().message` | 성공/실패 toast 표시 |

### 캐시 무효화

`useUpdateContent` mutation 성공 시 `queryClient.invalidateQueries({ queryKey: ['contents'] })`를
호출하여 콘텐츠 상세 + 목록 자동 갱신.

| **네트워크 오류** | Axios 인터셉터에서 공통 처리 (401 -> 로그인, 403 -> 권한 없음) |

| **수정 불가 상태** | 상세 페이지에서 수정 버튼이 DRAFT/REJECTED 외에는 비활성화 |

## 12. 사용 시나리오

### 시나리오 1: 초안 콘텐츠 수정
1. 콘텐츠 목록에서 DRAFT 상태 콘텐츠의 수정 아이콘 클릭
2. 수정 페이지 진입 -- 기존 제목, 본문, 플랫폼, 예약일시가 미리 채워짐
3. 본문 내용 수정
4. 플랫폼에 Facebook 추가 체크
5. [저장] 클릭 -- "콘텐츠가 수정되었습니다" 토스트
6. 상세 페이지로 자동 이동 -- 변경 내용 확인

### 시나리오 2: 반려 콘텐츠 수정 후 재요청
1. 콘텐츠 상세에서 REJECTED 상태 확인 + 반려 사유 확인
2. [수정] 버튼 클릭 -- 수정 화면 진입
3. 반려 사유에 맞춰 제목과 본문 수정
4. [저장] 클릭 -- 상세 페이지 복귀
5. 상태가 DRAFT로 변경됨
6. [검토 요청] 버튼으로 재검토 요청

### 시나리오 3: 예약 게시일시 변경
1. 수정 화면 진입 -- 기존 예약일시가 DatePicker에 표시
2. 날짜 선택기에서 새 일시로 변경
3. [저장] 클릭 -- 예약일시 업데이트
4. 상세 페이지에서 변경된 예약일시 확인

### 시나리오 4: 수정 취소
1. 수정 화면에서 내용 변경 후 마음이 바뀜
2. [취소] 버튼 클릭
3. 변경 사항 저장 없이 상세 페이지로 복귀
4. 기존 콘텐츠 데이터 유지

### 시나리오 5: 필수 항목 누락 시
1. 수정 화면에서 제목을 비움 (전체 삭제)
2. [저장] 클릭
3. Ant Design Form 유효성 검사 -- "제목을 입력하세요" 인라인 에러 표시
4. 저장되지 않고 해당 필드에 포커스

## 15. 새 콘텐츠 작성과의 차이점

| 항목 | 새 콘텐츠 작성 (Create) | 콘텐츠 수정 (Edit) |
|------|----------------------|-------------------|
| **경로** | `/contents/create` | `/contents/:id/edit` |
| **데이터 조회** | 없음 | `useContent(id)` 호출 |
| **폼 초기값** | 빈 폼 | 기존 데이터 프리필 |
| **API** | `POST /contents` | `PUT /contents/:id` |
| **Hook** | `useCreateContent()` | `useUpdateContent()` |
| **성공 후 이동** | `/contents` (목록) | `/contents/:id` (상세) |
| **뒤로가기 대상** | `/contents` (목록) | `/contents/:id` (상세) |
| **성공 메시지** | "콘텐츠가 작성되었습니다" | "콘텐츠가 수정되었습니다" |
| **실패 메시지** | "콘텐츠 작성에 실패했습니다" | "콘텐츠 수정에 실패했습니다" |
| **AI 패널** | 있음 (우측) | 없음 (Phase 1-B 예정) |
| **미리보기 패널** | 있음 (우측) | 없음 (Phase 2 예정) |
| **임시 저장** | 있음 | 없음 (이미 저장된 콘텐츠) |
| **검토 요청** | 있음 (헤더) | 없음 (상세에서 수행) |
| **워크플로우 바** | 있음 (하단) | 없음 |

# 댓글 관리 (CommentsListPage)

> **경로:** `/comments`
> **소스:** `frontend/src/features/comments/pages/CommentsListPage.tsx`
> **접근 권한:** AM (수탁업체 관리자), AO (수탁업체 실무자), CD (위탁기관 담당자)
> **Phase:** 1-B (MVP 확장)
> **관련 기능:** F04 댓글관리, F05 AI답글

---

## 1. 화면 개요

연동된 전체 SNS 채널(YouTube, Instagram, Facebook, X, 네이버 블로그)의 댓글을
하나의 통합 댓글함으로 수집·표시하고, 감성 분석 결과에 따라 필터링하며,
선택한 댓글에 대해 답글 게시·숨김 처리·삭제 승인 요청 등의 조치를 수행하는 화면이다.
AI 답글 생성 기능을 통해 댓글 대응 효율을 높일 수 있다.

### 핵심 원칙
- **Human-in-the-Loop:** AI가 생성한 답글은 "제안"이며, 최종 결정은 사용자가 한다
- **Fallback 필수:** AI 답글 생성 실패 시에도 수동 입력으로 답글 작성 가능
- **워크플로우 내장 AI:** AI 답글 기능이 별도 메뉴가 아닌 댓글 상세 패널 안에서 호출됨

| **작성자명** | `Text` (strong, text-sm) |
| **플랫폼 태그** | `Tag` (플랫폼별 색상, 약어 표시 — YT/IG/FB/X/Blog) |
| **시간** | `Text` (type="secondary", text-[11px]) — 상대 시간 표시 |
| **댓글 내용** | 한 줄 말줄임 (truncate, text-[13px]) |

### 6.2 선택 상태
| 항목 | 내용 |

### 6.3 시간 표시 로직 (formatTime)

| 조건 | 표시 |

## 9. 연결 흐름 (Connected Flows)

### 9.1 댓글 조회 → 답글 게시 흐름

```
[댓글 목록 진입] (/comments)
       │
       ↓
[감성 탭/플랫폼/검색 필터링]
       │
       ↓
[댓글 선택] → [우측 상세 패널 표시]
       │
       ├──→ [수동 답글 작성] → [답글 게시] → POST /comments/:id/reply
       │
       ├──→ [AI 답글 생성] → [모달: 제안 3개] → [사용하기] → replyText 적용
       │         │                                                │
       │         └──(실패)──→ "AI 답글 생성에 실패했습니다" toast    │
       │                                                         ↓
       │                                              [답글 게시] → API 호출
       │
       ├──→ [숨김 처리] → Popconfirm → POST /comments/:id/hide
       │
       └──→ [삭제 승인 요청] → Popconfirm → POST /comments/:id/delete-request
```

### 9.2 위험 댓글 연계 흐름

```
[댓글 관리] ──(🔴 위험 탭)──→ 위험 감성 댓글 필터링
       │
       └──→ [위험 댓글 관리] (/comments/dangerous) ← 사이드바 별도 메뉴
```

### 9.3 답글 템플릿 연계 흐름

```
[댓글 관리] ──(답글 작성 시)──→ 답글 템플릿 활용 가능
       │
       └──→ [답글 템플릿 관리] (/comments/reply-templates) ← 사이드바 별도 메뉴
```

## 10. API 엔드포인트 연결

| 동작 | HTTP | 엔드포인트 | Hook | Phase |

## 11. 데이터 타입

### 11.1 댓글 레코드 (CommentRecord)

```typescript
interface CommentRecord {
  id: string;                          // 댓글 고유 ID
  organization_id: string;             // 기관 ID (RLS)
  content_id: string | null;           // 원본 콘텐츠 ID
  channel_id: string;                  // 채널 ID
  platform: string;                    // 플랫폼 (YOUTUBE/INSTAGRAM/FACEBOOK/X/NAVER_BLOG)
  external_id: string;                 // 플랫폼 외부 ID
  text: string;                        // 댓글 본문
  author_name: string;                 // 작성자 이름
  author_profile_url: string | null;   // 작성자 프로필 URL
  parent_comment_id: string | null;    // 부모 댓글 ID (대댓글인 경우)
  sentiment: string | null;            // 감성 분석 결과 (POSITIVE/NEUTRAL/NEGATIVE/DANGEROUS)
  sentiment_confidence: number | null;  // 감성 분석 신뢰도 (0~1)
  dangerous_level: string | null;      // 위험 수준
  keywords: string[] | null;           // 추출 키워드 배열
  status: string;                      // 상태 (UNPROCESSED/PUBLISHED/HIDDEN/PENDING_DELETE/DELETED)
  reply_text: string | null;           // 게시된 답글 텍스트
  reply_draft: string | null;          // 답글 임시 저장 텍스트
  replied_at: string | null;           // 답글 게시 일시
  hidden_reason: string | null;        // 숨김 사유
  delete_reason: string | null;        // 삭제 사유
  processed_by: string | null;         // 처리자 ID
  platform_created_at: string | null;  // 플랫폼 원본 작성 일시
  created_at: string;                  // 시스템 생성 일시
  updated_at: string;                  // 시스템 수정 일시
}
```

### 11.2 AI 답글 생성 요청

```typescript
interface AiGenerateReplyRequest {
  comment_text: string;   // 원본 댓글 텍스트
```

## 14. 사용 시나리오

### 시나리오 1: 기본 댓글 확인 및 수동 답글
1. 사이드바 > 댓글 > 통합 댓글함 진입
2. 전체 댓글 목록에서 답변이 필요한 댓글 클릭
3. 우측 상세 패널에서 감성 분석 결과와 키워드 확인
4. 답변 내용 직접 입력
5. [답글 게시] 클릭 → API 호출 후 목록 갱신

### 시나리오 2: AI 답글 활용
1. 부정/위험 댓글 선택
2. [AI 답글 생성] 클릭
3. 모달에서 3개 제안 확인, 신뢰도 높은 답글 [사용하기] 클릭
4. 답글 입력란에 자동 적용, 필요시 내용 수정
5. [답글 게시] 클릭

### 시나리오 3: 감성 기반 필터링
1. [🔴 위험] 탭 클릭 → 위험 댓글만 표시
2. 플랫폼 필터에서 "YouTube" 선택 → YouTube 위험 댓글만 표시
3. 각 댓글에 대해 숨김 처리 또는 삭제 승인 요청

### 시나리오 4: 위험 댓글 긴급 대응
1. [🔴 위험] 탭에서 위험 댓글 확인
2. 댓글 선택 → 감성 신뢰도 95%로 확인
3. [숨김 처리] 클릭 → Popconfirm 확인 → 즉시 숨김
4. 필요시 [삭제 승인 요청] 클릭 → 관리자 승인 대기

### 시나리오 5: AI 실패 시 Fallback
1. [AI 답글 생성] 클릭 → AI 서비스 오류
2. "AI 답글 생성에 실패했습니다" toast 표시
3. 수동으로 답글 내용 직접 입력하여 게시 계속

## 16. 편차 사항 (설계 vs 구현)

### D1. 답글 템플릿 삽입 기능

설계서에는 §9.3에서 "답글 템플릿 활용 가능" 연계만 언급되었으나,
구현에서는 댓글 상세 패널 내에 `Dropdown` "템플릿 삽입" 버튼이 포함되어 있다.

- `useReplyTemplates()` 훅으로 활성 템플릿 목록 조회
- `CopyOutlined` 아이콘 + "템플릿 삽입" Dropdown
- 클릭 시 템플릿 content를 replyText에 직접 삽입
- 템플릿 없으면 버튼 비활성화

### D2. delete-request RBAC — AO 전용

설계서에는 "삭제 승인 요청"의 역할을 명시하지 않았으나,
구현에서는 AO만 삭제 요청 가능 (`require_roles(AGENCY_OPERATOR)`).
AM/CD는 `delete-approve` 엔드포인트에서 승인 처리.

> **워크플로우:** AO(실무자)가 삭제 요청 → AM/CD(관리자/담당자)가 승인

## 17. BE 보너스 기능 (문서 대비 추가 구현)

### B1. 답글 템플릿 Dropdown (워크플로우 내장)

AI 답글 버튼 옆에 "템플릿 삽입" Dropdown을 배치하여,
사전 정의된 답글 템플릿을 한 번의 클릭으로 답글 입력란에 삽입 가능.

### B2. 댓글 상태 전이 검증 (State Machine)

```python
_VALID_TRANSITIONS = {
    UNPROCESSED: {PUBLISHED, HIDDEN, PENDING_DELETE},
    PUBLISHED:   {HIDDEN, PENDING_DELETE},
    HIDDEN:      {UNPROCESSED, PENDING_DELETE},
    PENDING_DELETE: {DELETED, UNPROCESSED},
    DELETED:     {},  # 종료 상태
}
```

잘못된 상태 전이 시 `WorkflowStateConflictError` 발생.

### B3. 플랫폼 어댑터 Best-effort 연동

답글/숨김/삭제 시 실제 SNS 플랫폼에 API 호출 시도.
실패해도 DB 상태는 변경하여 서비스 중단 방지.

### B4. Meilisearch 인덱싱

답글 게시 시 `_index_comment_to_search()`로 검색 인덱스 동기화 (best-effort).

### B5. 위험 댓글 Partial Index

`idx_comments_dangerous` — `sentiment='DANGEROUS'`인 행만 인덱싱하여
위험 댓글 전용 쿼리 성능 최적화.

### B6. 구조화 로깅 (structlog)

모든 action에 구조화 로깅:
- `comment_replied`, `comment_hidden`, `comment_delete_requested`
- `comment_delete_approved`, `comment_danger_ignored`
- `platform_reply_failed`, `platform_hide_failed`, `platform_delete_failed`

### B7. RLS + Unique Constraint

- `comments`, `reply_templates` 모두 tenant_isolation + sa_bypass
- `uq_comments_channel_external` — 채널+외부ID 중복 수집 방지

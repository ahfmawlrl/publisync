# 콘텐츠 상세 (ContentDetailPage)

> **경로:** `/contents/:id`
> **소스:** `frontend/src/features/contents/pages/ContentDetailPage.tsx`
> **접근 권한:** AM (수탁업체 관리자), AO (수탁업체 실무자), CD (위탁기관 담당자)
> **Phase:** 1-A (최소 MVP)
> **관련 기능:** F01 통합게시, F09 승인워크플로우

---

## 1. 화면 개요

특정 콘텐츠의 상세 정보를 조회하는 화면이다. 콘텐츠 본문, 플랫폼별 미리보기,
게시 이력, 승인 이력을 4개 탭으로 분리하여 제공하며, 우측 사이드 패널에 요약 정보를 표시한다.
콘텐츠 상태에 따라 헤더에 수정/검토 요청 버튼이 조건부로 활성화된다.

### 핵심 원칙
- **정보 가시성:** 콘텐츠의 전체 상태와 이력을 한 화면에서 파악 가능
- **상태 기반 액션:** 현재 상태에 따라 수행 가능한 액션만 표시
- **플랫폼별 미리보기:** 각 플랫폼에 게시될 형태를 사전 확인

| 항목 | 내용 |
|------|------|

#### 타임라인 항목 구성

| 요소 | 설명 |
|------|------|

| **날짜** | `created_at` 한국어 로케일 (text-xs, text-gray-400) |

| **승인 완료** | `green` | `APPROVED`, `PUBLISHED`, `SCHEDULED` | "승인 완료" 텍스트 |
| **반려됨** | `red` | `REJECTED` | "반려됨" 텍스트 |
| **게시 완료** | `green` | `PUBLISHED` | "게시 완료" 텍스트 |

## 7. 연결 흐름 (Connected Flows)

### 7.1 상세 화면 진입/이탈 흐름

```
[콘텐츠 목록] ──(제목 링크/상세 버튼)──→ [콘텐츠 상세]
                                              │
                                        ┌─────┴──────┐
                                        ↓            ↓
                                   [← 목록 복귀]  [수정 버튼]
                                   (/contents)    (/contents/:id/edit)
                                                  (DRAFT/REJECTED만)
```

### 7.2 검토 요청 흐름

```
[콘텐츠 상세] (status: DRAFT)
       │
       └──(검토 요청 버튼)──→ POST /contents/:id/request-review
                                   │
                             ┌─────┴──────┐
                             ↓            ↓
                        [성공: 토스트]  [실패: 토스트]
                        상태 → PENDING   상태 유지
                        + 캐시 무효화
                             │
                             ↓
                      [승인 대기 목록] (/approvals)에 표시
                             │
                       ┌─────┴──────┐
                       ↓            ↓
                  [CD: 승인]    [CD: 반려]
                       │            │
                       ↓            ↓
                 [자동 게시]   [수정 가능 상태로 복귀]
```

### 7.3 게시 이력 조회 흐름

```
[게시 이력 탭 클릭] ──→ GET /contents/:id/publish-history
                              │
                        ┌─────┴──────┐
                        ↓            ↓
                   [이력 있음]   [이력 없음]
                   Timeline      Empty 표시
                   표시
```

## 8. API 엔드포인트 연결

| 동작 | HTTP | 엔드포인트 | Hook | Phase |
|------|------|-----------|------|-------|
| 콘텐츠 상세 조회 | GET | `/contents/:id` | `useContent()` | 1-A |
| 게시 이력 조회 | GET | `/contents/:id/publish-history` | `usePublishHistory()` | 1-A |
| 검토 요청 | POST | `/contents/:id/request-review` | `useRequestReview()` | 1-A |

### 8.1 상세 조회 API

| 항목 | 내용 |
|------|------|
| **응답 형태** | `ApiResponse<ContentRecord>` |
| **캐시 키** | `['contents', contentId]` |
| **활성 조건** | `enabled: !!contentId` -- ID 없으면 비활성 |

### 8.2 게시 이력 API

| 항목 | 내용 |
|------|------|
| **응답 형태** | `PaginatedResponse<PublishResultRecord>` |
| **캐시 키** | `['contents', contentId, 'publish-history', { page }]` |
| **파라미터** | `page` (기본 1), `limit` (기본 50) |
| **활성 조건** | `enabled: !!contentId` |

## 9. 데이터 타입

### 9.1 콘텐츠 레코드 (ContentRecord)

```typescript
interface ContentRecord {
  id: string;                                  // 콘텐츠 고유 ID (UUID)
  organization_id: string;                     // 소속 기관 ID
  title: string;                               // 콘텐츠 제목
  body: string | null;                         // 본문/설명문
  status: string;                              // 콘텐츠 상태 (12가지)
  platforms: string[];                         // 게시 대상 플랫폼 배열
  channel_ids: string[];                       // 채널 ID 배열
  scheduled_at: string | null;                 // 예약 게시 일시 (ISO 8601)
  author_id: string;                           // 작성자 ID
  platform_contents: Record<string, unknown> | null;  // 플랫폼별 맞춤 콘텐츠
  metadata: Record<string, unknown> | null;    // 추가 메타데이터
  ai_generated: boolean;                       // AI 제안 사용 여부
  media_urls: string[];                        // 미디어 파일 URL 배열
  created_at: string;                          // 작성일 (ISO 8601)
  updated_at: string;                          // 수정일 (ISO 8601)
}
```

### 9.2 게시 결과 레코드 (PublishResultRecord)

```typescript
interface PublishResultRecord {
  id: string;                        // 게시 결과 고유 ID
  content_id: string;                // 콘텐츠 ID
  channel_id: string;                // 채널 ID
  status: string;                    // 게시 상태 (SUCCESS/FAILED/PENDING 등)
  platform_post_id: string | null;   // 플랫폼 게시물 ID
  platform_url: string | null;       // 게시된 URL
  error_message: string | null;      // 오류 메시지
  retry_count: number;               // 재시도 횟수
  views: number;                     // 조회수
  likes: number;                     // 좋아요 수
  shares: number;                    // 공유 수
  comments_count: number;            // 댓글 수
  created_at: string;                // 생성일 (ISO 8601)
}
```

| 영역 | 도구 | 설명 |
|------|------|------|
| **URL 파라미터** | `useParams()` | `:id` -- 콘텐츠 ID 추출 |
| **콘텐츠 데이터** | TanStack Query `useContent()` | 콘텐츠 상세 조회 (캐시 키: `['contents', id]`) |
| **게시 이력** | TanStack Query `usePublishHistory()` | 게시 결과 목록 조회 |
| **검토 요청** | TanStack Query `useRequestReview()` | 검토 요청 mutation |
| **네비게이션** | `useNavigate()` | 목록/수정 화면 이동 |
| **메시지** | `App.useApp().message` | 성공/실패 toast 표시 |

### 캐시 무효화

`useRequestReview` mutation 성공 시 `queryClient.invalidateQueries({ queryKey: ['contents'] })`를
호출하여 콘텐츠 상세 + 목록 자동 갱신.

| 상황 | 처리 |

## 12. 사용 시나리오

### 시나리오 1: 콘텐츠 상세 조회
1. 콘텐츠 목록에서 제목 클릭 -- 상세 페이지 진입
2. "콘텐츠 내용" 탭에서 본문 확인
3. 정보 카드에서 플랫폼, 예약일시, AI 생성 여부 확인
4. [<-] 버튼으로 목록 복귀

### 시나리오 2: 플랫폼별 미리보기 확인
1. 상세 페이지에서 "플랫폼별 미리보기" 탭 클릭
2. YouTube, Instagram, Facebook 등 선택된 플랫폼별 카드 확인
3. 각 플랫폼에 맞춰 잘린 본문과 제목 확인
4. 수정이 필요하면 [수정] 버튼 클릭 -- 수정 화면 이동

### 시나리오 3: 초안 콘텐츠 검토 요청
1. DRAFT 상태 콘텐츠 상세 진입
2. 본문과 플랫폼 설정 최종 확인
3. [검토 요청] 버튼 클릭
4. "검토 요청 완료" 토스트 확인
5. 상태 태그가 "검토 대기"로 변경됨
6. 수정/검토 요청 버튼 사라짐

### 시나리오 4: 게시 이력 확인
1. PUBLISHED 상태 콘텐츠 상세 진입
2. "게시 이력" 탭 클릭
3. Timeline에서 각 플랫폼별 게시 성공/실패 확인
4. SUCCESS인 항목의 "링크" 클릭 -- 실제 게시된 페이지 새 탭으로 열기
5. FAILED인 항목의 오류 메시지 확인

### 시나리오 5: 승인 이력 확인
1. 콘텐츠 상세에서 "승인 이력" 탭 클릭
2. Timeline에서 작성 -> 검토 요청 -> 승인/반려 -> 게시 흐름 확인
3. 각 단계별 색상으로 진행 상태 파악

### 시나리오 6: 반려 콘텐츠 수정 후 재요청
1. REJECTED 상태 콘텐츠 상세 진입
2. 승인 이력 탭에서 반려 사유 확인
3. [수정] 버튼 클릭 -- 수정 화면 이동
4. 내용 수정 후 저장 -- 상세 페이지 복귀
5. 상태가 DRAFT로 변경 후 다시 [검토 요청] 가능

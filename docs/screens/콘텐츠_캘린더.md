# 콘텐츠 캘린더 (CalendarPage)

> **경로:** `/calendar`
> **소스:** `frontend/src/features/calendar/pages/CalendarPage.tsx`
> **접근 권한:** AM (수탁업체 관리자), AO (수탁업체 실무자), CD (위탁기관 담당자)
> **Phase:** 2
> **관련 기능:** F10 콘텐츠 캘린더

---

## 1. 화면 개요

수탁업체 실무자(AO), 관리자(AM), 위탁기관 담당자(CD)가 예약 게시 일정, 공휴일, 기념일, 사용자 정의 이벤트를
FullCalendar 기반 캘린더 뷰에서 시각적으로 확인하고 관리하는 화면이다. 월간/주간/일간 뷰 전환, 이벤트 타입별
필터링, 드래그 앤 드롭 일정 변경, 공휴일 관리(추가/삭제/반복 설정) 기능을 제공한다.
날짜 클릭 시 해당 날짜로 예약된 새 콘텐츠 작성 화면으로 이동한다.

### 핵심 원칙
- **시각적 일정 관리:** FullCalendar로 게시 일정을 한눈에 파악
- **드래그 앤 드롭 일정 변경:** 캘린더에서 이벤트를 드래그하여 직관적으로 일정 변경
- **이벤트 타입별 색상 구분:** 예약 게시(파랑), 공휴일(빨강), 기념일(금색), 사용자 정의(보라/초록)

> **D1 참고:** 사용자 정의 이벤트는 Tag 색상은 `purple`, FullCalendar 배경색은 `#52c41a` (초록)으로
> 각각 다른 컨텍스트에서 식별성을 확보한다.

|------|------|
| **트리거** | `dateClick` 콜백 |

#### 이벤트 드롭 (handleEventDrop) — 드래그 앤 드롭 일정 변경

| 항목 | 내용 |
|------|------|
| **트리거** | `eventDrop` 콜백 (이벤트를 다른 날짜로 드래그) |
| **동작** | PATCH `/calendar/events/:id/reschedule` 호출 |

| **실패** | "일정 변경에 실패했습니다" toast + `info.revert()` (원위치 복귀) |

#### 날짜 범위 변경 (handleDatesSet)

| 항목 | 내용 |
|------|------|
| **트리거** | `datesSet` 콜백 (뷰 변경, 페이지 이동 시) |
| **동작** | 보이는 범위의 중간점을 `currentDate`로 설정 → API 조회 범위 갱신 |
| **조회 범위** | `currentDate` 기준 해당 월의 시작 -7일 ~ 끝 +7일 |

## 8. 연결 흐름 (Connected Flows)

### 8.1 캘린더 탐색 흐름

```
[캘린더 탭]
       │
       ├──(이전/다음/오늘 버튼)──→ datesSet → currentDate 갱신 → API 재조회
       │
       ├──(월간/주간/일간 전환)──→ 뷰 전환 + datesSet → 범위 재계산
       │
       └──(이벤트 타입 필터)──→ eventTypeFilter 변경 → API 재조회
```

### 8.2 이벤트 상호작용 흐름

```
[캘린더의 이벤트]
       │
       ├──(이벤트 클릭)──→ [일정 상세 모달]
       │                      │
       │                ┌─────┴──────────┐
       │                ↓                ↓
       │           [일정 변경]      [콘텐츠 보기]
       │                │                │
       │                ↓                ↓
       │        [일정 변경 모달]   navigate('/contents/:id')
       │         (날짜/시간 변경)
       │                │
       │                ↓
       │        PATCH /calendar/events/:id/reschedule
       │
       ├──(이벤트 드래그 앤 드롭)──→ PATCH /calendar/events/:id/reschedule
       │                              │
       │                        ┌─────┴──────┐
       │                        ↓            ↓
       │                   [성공 toast]  [실패 toast + 원위치 복귀]
       │
       └──(날짜 클릭)──→ navigate('/contents/create?date=YYYY-MM-DD')
```

### 8.3 공휴일 관리 흐름

```
[공휴일 관리 탭]
       │
       ├──(연도 선택)──→ holidayYear 변경 → GET /calendar/holidays?year=YYYY
       │
       ├──(공휴일 추가)──→ [공휴일 추가 모달]
       │                      │
       │                      ↓
       │               폼 입력 → PUT /calendar/holidays (기존 + 신규)
       │                      │
       │                      ↓
       │               "공휴일이 추가되었습니다" toast
       │
       └──(공휴일 삭제)──→ PUT /calendar/holidays (삭제 대상 제외)
                              │
                              ↓
                       "공휴일이 삭제되었습니다" toast
```

### 10.1 캘린더 이벤트 (CalendarEvent)


## 13. 사용 시나리오

### 시나리오 1: 월간 캘린더에서 일정 확인
1. 사이드바 > 캘린더 클릭
2. 월간 뷰에서 색상별 이벤트 확인 (파랑=예약 게시, 빨강=공휴일, 금색=기념일)
3. 이벤트 타입 필터에서 "예약 게시" 선택 → 예약 게시 이벤트만 표시
4. 이벤트 클릭 → 상세 모달에서 제목, 타입, 날짜, 플랫폼, 상태 확인
5. [콘텐츠 보기] 클릭 → 해당 콘텐츠 상세 페이지로 이동

### 시나리오 2: 드래그 앤 드롭으로 일정 변경
1. 캘린더에서 예약 게시 이벤트를 다른 날짜로 드래그
2. "'서울시 봄 행사 안내' 일정이 2024-03-15로 변경되었습니다" toast 확인
3. 실패 시 이벤트가 자동으로 원래 위치로 복귀

### 시나리오 3: 모달을 통한 일정 변경
1. 이벤트 클릭 → 상세 모달
2. [일정 변경] 클릭 → 일정 변경 모달
3. 새로운 날짜와 예약 시간 선택
4. [변경] 클릭 → "일정이 변경되었습니다" toast

### 시나리오 4: 공휴일 관리
1. [공휴일 관리] 탭 클릭
2. 연도 선택기에서 2024년 선택
3. [공휴일 추가] 클릭 → 모달 열림
4. 공휴일명: "개천절", 날짜: 10월 3일, 매년 반복: "매년 반복" 선택
5. [추가] 클릭 → 테이블에 새 공휴일 표시 + 캘린더에 빨간색 이벤트 표시
6. 불필요한 공휴일의 삭제 버튼 클릭 → 즉시 삭제

### 시나리오 5: 날짜 클릭으로 새 콘텐츠 작성
1. 캘린더에서 빈 날짜 클릭
2. 자동으로 `/contents/create?date=2024-03-20` 으로 이동
3. 해당 날짜에 예약 게시할 콘텐츠 작성 시작

## 15. BE 보너스 기능 (문서 대비 추가 구현)

### B1. 일정 충돌 감지 (Conflict Detection)

BE `CalendarService.reschedule_event()`에서 일정 변경 시 같은 시간에 3개 이상 이벤트가
존재하면 `meta.conflict` 경고를 응답에 포함한다.

```python
# CalendarService.detect_conflicts()
count = await self._repo.count_events_at_time(org_id, scheduled_at)
if count >= 3:
    return {"count": count, "warning": "해당 시간에 이미 N개의 이벤트가 예약되어 있습니다."}
```

> FE에서는 아직 `meta.conflict`를 활용하지 않으나, 향후 경고 Alert 표시에 사용 가능.

### B2. Content ↔ Calendar 동기화 헬퍼

콘텐츠 예약 게시/변경/삭제 시 캘린더 이벤트를 자동으로 생성/수정/삭제하는 3개 서비스 헬퍼:

| 메서드 | 트리거 | 동작 |
|--------|--------|------|
| `create_content_event()` | 콘텐츠 예약 게시 | SCHEDULED_POST 이벤트 생성 (content_id FK 연결) |
| `update_content_event()` | 콘텐츠 일정 변경 | content_id로 이벤트 조회 후 title/scheduled_at 수정 |
| `delete_content_events()` | 콘텐츠 삭제/취소 | content_id로 연결 이벤트 삭제 |

> DB FK: `content_id → contents.id (ondelete=SET NULL)` — 콘텐츠 삭제 시에도 안전.

### B3. 구조화 로깅 (structlog)

모든 mutation 작업에 구조화 로깅 포함:
- `calendar_event_rescheduled` — 일정 변경 추적
- `calendar_holidays_deleted` / `calendar_holidays_updated` — 공휴일 벌크 작업 추적
- `calendar_content_event_created` / `_updated` / `_deleted` — 콘텐츠 동기화 추적

### B4. DB 복합 인덱스

`idx_calendar_events_org_date` (organization_id + event_date) 복합 인덱스로
조직별 날짜 범위 조회(가장 빈번한 쿼리 패턴) 성능을 최적화한다.

### B5. RLS 정책 (Migration 011)

```sql
-- tenant_isolation: 조직별 데이터 격리
CREATE POLICY tenant_isolation ON calendar_events
    USING (organization_id = current_setting('app.current_org_id')::uuid);

-- sa_bypass: 시스템 관리자 전체 접근
CREATE POLICY sa_bypass ON calendar_events
    USING (current_setting('app.user_role', true) = 'SYSTEM_ADMIN');
```

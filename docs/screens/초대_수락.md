# 초대 수락 (InvitePage)

> **경로:** `/invite?token=xxx`
> **소스:** `frontend/src/features/auth/pages/InvitePage.tsx`
> **접근 권한:** 공개 (비인증 사용자, 유효한 초대 토큰 필요)
> **Phase:** 1-A (최소 MVP)
> **관련 기능:** F08 권한관리 (사용자 초대, 계정 생성, RBAC 역할 할당)

---

## 1. 화면 개요

수탁업체 관리자(AM) 또는 시스템 관리자(SA)가 발송한 초대 이메일의 링크를 통해 접근하는 화면이다.
초대 토큰을 서버에 검증한 뒤, 초대된 기관명/이메일/역할을 확인하고 이름과 비밀번호를 설정하여 가입을 완료한다.
가입 완료 시 자동으로 로그인되어 대시보드로 이동한다.

### 핵심 원칙
- **토큰 기반 접근:** URL의 `?token=` 파라미터가 유효해야만 가입 폼 표시
- **초대 정보 확인:** 기관명, 이메일, 역할을 사용자에게 명확히 표시
- **자동 로그인:** 가입 완료 시 별도 로그인 없이 바로 서비스 이용 가능

## 3. 토큰 검증 및 화면 분기

| 상태 | 조건 | 표시 |
|------|------|------|
| **토큰 없음** | `token === null` | `Result` (status="error") — "잘못된 초대 링크" + 로그인 링크 |
| **토큰 검증 중** | `isLoading === true` | `Spin` (size="large") — 전체 화면 중앙 |
| **토큰 검증 실패** | `isError === true` 또는 `!invite` | `Result` (status="warning") — "초대가 만료되었거나 유효하지 않습니다" + 로그인 링크 |
| **토큰 유효** | `invite` 데이터 존재 | 가입 폼 표시 |

### 토큰 검증 API

| 항목 | 내용 |
|------|------|
| **Hook** | `useInviteVerify(token)` |
| **방식** | TanStack Query `useQuery` |
| **queryKey** | `['invite', 'verify', token]` |
| **API** | `GET /auth/invite/verify?token=xxx` |
| **활성화 조건** | `enabled: !!token` — 토큰이 존재할 때만 요청 |
| **반환 데이터** | `InviteVerifyResponse` — 이메일, 역할, 기관명, 만료일시 |

## 7. 연결 흐름 (Connected Flows)

### 7.1 초대 발송 → 수락 → 로그인 전체 흐름

```
[AM/SA: 사용자 초대] ──(POST /users/invite)──→ 초대 이메일 발송
       │
       ↓
[사용자: 이메일 수신] ──(초대 링크 클릭)──→ /invite?token=xxx
       │
       ↓
[토큰 검증] ──(GET /auth/invite/verify)
       │
       ├──(유효)──→ [가입 폼 표시]
       │                 │
       │           (이름 + 비밀번호 입력)
       │                 │
       │                 ↓
       │      POST /auth/invite/accept
       │                 │
       │                 ├──(성공)──→ Zustand setAuth(tokens, user)
       │                 │                 │
       │                 │                 ↓
       │                 │           message.success('가입이 완료되었습니다')
       │                 │                 │
       │                 │                 ↓
       │                 │           navigate('/') → [대시보드]
       │                 │
       │                 └──(실패)──→ "가입에 실패했습니다. 다시 시도하세요." 오류
       │
       ├──(토큰 없음)──→ Result(error) "잘못된 초대 링크"
       │
       └──(토큰 만료)──→ Result(warning) "초대가 만료되었거나 유효하지 않습니다"
```

### 7.2 자동 로그인 흐름

```
[가입 완료 응답] ──→ LoginResponse (tokens + user)
       │
       ↓
useAuthStore.setAuth({
  accessToken: data.tokens.access_token,
  refreshToken: data.tokens.refresh_token
}, {
  id, email, name, role, status
})
       │
       ↓
localStorage 'publisync-auth'에 persist
       │
       ↓
navigate('/') → [대시보드] (별도 로그인 불필요)
```

## 9. 데이터 타입

### 9.1 초대 검증 응답 (InviteVerifyResponse)

```typescript
interface InviteVerifyResponse {
  email: string;              // 초대된 이메일 주소
  role: string;               // 할당된 역할 (SA|AM|AO|CD)
  organization_name: string;  // 초대한 기관명
  expires_at: string;         // 초대 만료 일시 (ISO 8601)
}
```

### 9.2 초대 수락 요청 (InviteAcceptRequest)

```typescript
interface InviteAcceptRequest {
  token: string;     // 초대 토큰 (URL 파라미터에서 추출)
  name: string;      // 사용자 이름
  password: string;  // 비밀번호 (8자 이상)
}
```

### 9.3 초대 수락 응답 (LoginResponse 재사용)

```typescript
// 초대 수락 API는 로그인과 동일한 응답 구조를 반환한다
interface LoginResponse {
  tokens: TokenResponse;  // JWT 토큰 쌍
  user: UserResponse;     // 사용자 정보
}

interface TokenResponse {
  access_token: string;   // JWT 액세스 토큰 (30분)
  refresh_token: string;  // JWT 리프레시 토큰 (7일)
  token_type: string;     // "bearer"
}

interface UserResponse {
  id: string;                     // 사용자 UUID
  email: string;                  // 이메일
  name: string;                   // 이름
  role: string;                   // RBAC 역할 (SA|AM|AO|CD)
  status: string;                 // 계정 상태
  profile_image_url: string | null; // 프로필 이미지 URL
}
```

### 9.4 가입 폼 데이터 (내부)

```typescript
// InvitePage 내부에서만 사용, confirm_password는 서버에 전송하지 않음
interface InviteFormData {
  name: string;              // 이름 (필수)
  password: string;          // 비밀번호 (필수, 8자 이상)
  confirm_password: string;  // 비밀번호 확인 (클라이언트 검증 전용)
}
```

## 12. 사용 시나리오

### 시나리오 1: 정상 초대 수락 (신규 사용자)

1. AM이 관리 화면에서 `operator@agency.kr`에게 AO 역할로 초대 발송
2. 사용자가 이메일의 초대 링크 클릭 → `/invite?token=xxx` 이동
3. 토큰 검증 성공 → "한국관광공사에서 초대했습니다" / "operator@agency.kr / AO" 표시
4. 이름 입력: "김실무"
5. 비밀번호 입력: "securepass123"
6. 비밀번호 확인: "securepass123"
7. [가입 및 시작하기] 클릭
8. "가입이 완료되었습니다" 메시지 → 자동 로그인 → 대시보드(`/`) 이동

### 시나리오 2: 만료된 초대 링크 접근

1. 초대 이메일을 며칠 후에 열어 링크 클릭
2. `/invite?token=expired_xxx` 이동
3. 토큰 검증 실패
4. "초대가 만료되었거나 유효하지 않습니다" 경고 표시
5. "로그인" 링크를 통해 로그인 화면으로 이동
6. AM에게 재초대 요청 필요

### 시나리오 3: 잘못된 URL (토큰 없음)

1. `/invite` (토큰 없이) 직접 접근
2. "잘못된 초대 링크" 오류 표시
3. "로그인" 링크 클릭 → `/login` 이동

### 시나리오 4: 가입 처리 실패

1. 토큰 검증 성공, 가입 폼 표시
2. 이름, 비밀번호 입력 후 [가입 및 시작하기] 클릭
3. 서버 오류 발생
4. "가입에 실패했습니다. 다시 시도하세요." 오류 메시지 표시
5. 동일한 폼에서 재시도 가능

## 14. 보안 관련 사항

| 항목 | 설명 |
|------|------|
| **초대 토큰** | 서버에서 생성한 일회성 토큰, URL 쿼리 파라미터로 전달 |
| **토큰 만료** | `expires_at` 필드로 서버 측 유효 시간 관리 |
| **비밀번호 최소 길이** | 8자 이상 (클라이언트 + 서버 양쪽 검증) |
| **confirm_password** | 서버에 전송하지 않음, 클라이언트 측 일치 검증만 수행 |
| **자동 로그인** | 가입 응답에 JWT 토큰 포함, 별도 로그인 단계 생략 |
| **이메일 변경 불가** | 초대된 이메일은 서버에서 결정, 사용자가 변경할 수 없음 |
| **역할 변경 불가** | 초대 시 지정된 역할 그대로 적용, 사용자 선택 불가 |

## 1. 문서 개요

### 1.1 목적

본 문서는 PubliSync 시스템을 구성하는 컨테이너(서비스)들이 **어떻게 조합되어 하나의 시스템을 이루고, 요청이 시스템 내부를 어떻게 통과하는가**를 정의한다.

기술 스택 결정서가 "무엇을 쓸 것인가"를 결정했다면, 본 문서는 "선택된 65개 기술이 **어떻게 연결되고 상호작용하는가**"를 설명한다.

### 1.2 대상 독자

| 독자 | 활용 방식 |
|---|---|
| 백엔드 개발자 | 레이어 구조, 미들웨어 체인, Celery 큐 설계를 기준으로 구현 |
| 프론트엔드 개발자 | 모듈 구조, 상태 관리 전략, API 통신 패턴을 기준으로 구현 |
| DevOps / 인프라 | Docker Compose 토폴로지, CI/CD 파이프라인, 모니터링 설계 참조 |
| 프로젝트 관리자 | Phase별 아키텍처 진화 경로, 기술 리스크 파악 |
| 보안 담당자 | Defense in Depth 계층, 암호화 전략, 감사 로그 설계 검토 |

### 1.3 문서 간 관계

## 2. 아키텍처 원칙

본 시스템의 모든 설계 결정은 다음 6개 원칙을 기준으로 판단한다.

| # | 원칙 | 설명 | 위반 시 영향 |
|:---:|---|---|---|
| 1 | **멀티테넌트 격리 우선** | 위탁기관 간 데이터는 DB(RLS), API(X-Workspace-Id), 파일(org_id 프리픽스), 캐시(키에 org_id), Celery(인자), SSE(채널명) 등 **모든 계층에서 격리**한다. | 타 기관 데이터 노출 — 법적 리스크 |
| 2 | **Human-in-the-Loop** | AI의 모든 산출물은 "제안(Suggestion)"이며, 최종 결정·게시 권한은 사람에게 있다. AI가 자동으로 외부에 콘텐츠를 게시하는 것은 **허용하지 않는다**. | AI 오류가 공공기관 명의로 외부 노출 |
| 3 | **비동기 분리 (10초 기준)** | 응답 시간이 10초를 초과할 수 있는 작업(AI 생성, 영상 처리, 다중 플랫폼 게시)은 반드시 **Celery 비동기 작업으로 분리**하고, 프론트엔드에서 진행 상태를 표시한다. | API 타임아웃, 사용자 UX 저하 |
| 4 | **Fallback 필수** | 모든 AI 기능, 외부 API 연동은 장애·타임아웃·품질 미달 시 **수동 대체 경로**를 반드시 제공한다. AI 없이도 핵심 업무(게시, 댓글 관리, 승인)가 동작해야 한다. | AI/외부 장애가 전체 서비스 중단으로 확대 |
| 5 | **환경변수 기반 설정** | DB 연결 문자열, Redis URL, API 키, 암호화 키 등 모든 환경 의존 설정은 **코드가 아닌 환경 변수**로 관리한다. Phase 전환 시 코드 변경 없이 인프라만 교체 가능해야 한다. | Phase 전환 시 코드 수정 필요, 보안 키 노출 |
| 6 | **Phase 점진 확장** | Phase 1-A에서 최소 컨테이너로 시작하고, Phase별로 서비스를 추가한다. 초기 아키텍처가 이후 Phase의 추가를 **구조적으로 방해하지 않아야** 한다. | 기능 추가 시 대규모 리팩토링 필요 |

## 3. 시스템 전체 구조

### 3.1 C4 Context 다이어그램

시스템 외부에서 PubliSync를 바라본 관점이다.

```
                          ┌───────────────────────┐
                          │    소셜 미디어 플랫폼   │
                          │  YouTube · Instagram   │
                          │  Facebook · X · Blog   │
                          └───────────┬───────────┘
                                      │ OAuth 2.0 / REST API
                                      │
┌─────────────┐   HTTPS    ┌──────────▼──────────┐   REST API   ┌─────────────┐
│ 수탁업체     │──────────▶│                      │◀────────────│ AI 서비스    │
│ 관리자/실무자 │           │                      │             │ (OpenAI 등)  │
└─────────────┘           │     PubliSync        │             └─────────────┘
                          │                      │
┌─────────────┐   HTTPS   │   공공기관 SNS 통합   │   SMTP       ┌─────────────┐
│ 위탁기관     │──────────▶│   관리 플랫폼         │─────────────▶│ 이메일 서버  │
│ 담당자       │           │                      │             └─────────────┘
└─────────────┘           │                      │
                          │                      │   Telegram API ┌─────────────┐
┌─────────────┐   HTTPS   │                      │───────────────▶│ Telegram Bot │
│ 시스템 관리자 │──────────▶│                      │               └─────────────┘
└─────────────┘           └──────────────────────┘
```

**외부 액터:**
- **수탁업체 관리자/실무자** — 콘텐츠 제작·게시, 댓글 관리, 리포트 생성
- **위탁기관 담당자** — 콘텐츠 검수·승인, 위험 댓글 대응 승인
- **시스템 관리자** — 수탁업체 계정 발급, 시스템 설정

**외부 시스템:**
- **소셜 미디어 플랫폼** — 게시, 댓글 수집, 분석 데이터 조회 (OAuth 2.0)
- **AI 서비스** — 메타데이터 생성, 감성 분석, 자막 생성, 리포트 초안 (litellm 경유)
- **이메일 서버** — 초대, 비밀번호 재설정, 알림 발송 (SMTP)
- **Telegram Bot** — 긴급 알림 발송 (Telegram Bot API)

## 4. 프론트엔드 아키텍처

### 4.1 모듈 구조

features/ 기반 기능 모듈 분해를 채택한다. 각 feature 모듈은 해당 기능에 필요한 컴포넌트, 훅, 타입, API 호출을 **자급자족**하며, 공유 레이어에만 의존한다.

```
src/
├── app/                          ← 앱 진입점, 프로바이더 래핑, 글로벌 설정
│   ├── App.tsx                   ← RouterProvider + QueryClientProvider + ConfigProvider
│   ├── router.tsx                ← 라우트 정의 (lazy import)
│   └── providers/                ← ThemeProvider, AuthProvider, WorkspaceProvider
│
├── shared/                       ← 공유 레이어 (기능 모듈이 의존)
│   ├── api/                      ← Axios 인스턴스, 인터셉터, 에러 핸들링
│   │   ├── client.ts             ← Axios 인스턴스 (baseURL, interceptors)
│   │   └── types.ts              ← ApiResponse<T>, PaginatedResponse<T>
│   ├── components/               ← 범용 UI 컴포넌트 (PageHeader, StatusBadge 등)
│   ├── hooks/                    ← 범용 훅 (useAuth, useWorkspace, usePermission)
│   ├── layouts/                  ← GlobalLayout, Sidebar, TopBar
│   ├── stores/                   ← Zustand 스토어 (UI 전용 상태)
│   ├── types/                    ← 공용 타입 (User, Organization, Role 등)
│   └── utils/                    ← 유틸리티 (dayjs 날짜 포맷, 권한 체크 등)
│
├── features/                     ← 기능별 모듈 (도메인 경계 일치)
│   ├── auth/                     ← 로그인, 토큰 갱신, 비밀번호 재설정
│   │   ├── components/
│   │   ├── hooks/                ← useLogin, useRefreshToken
│   │   ├── pages/                ← LoginPage, ResetPasswordPage
│   │   └── types.ts
│   ├── dashboard/                ← 대시보드 (F06)

**의존 방향 규칙:**

```

**TanStack Query 글로벌 설정:**

```

└── refetchOnReconnect: true     ← 네트워크 복구 시 자동 갱신

#### Axios 인터셉터 체인

```

┌──────────┐    ┌──────────────┐    ┌──────────────┐    ┌────────┐

└──────────┘    └──────────────┘    └──────────────┘    └────────┘
```

**JWT 자동 갱신 흐름:**

```
1. API 호출 → 401 Unauthorized 수신
2. refreshToken으로 POST /auth/refresh 호출
3. 성공 → 새 accessToken 저장 → 원본 요청 재시도
4. 실패 → 로그아웃 처리 → 로그인 페이지로 리다이렉트
5. 동시 요청 경쟁 방지: 갱신 중인 Promise를 공유하여 중복 갱신 차단
```

## 5. 백엔드 아키텍처

### 5.1 레이어 구조

4-Layer 아키텍처를 채택한다. 각 레이어는 하위 레이어에만 의존하며, 상위 레이어를 직접 참조하지 않는다.

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 1: Router (API 레이어)                                │
│  ─────────────────────────────────────────────────────────── │
│  • HTTP 요청 수신, 경로 매핑                                  │
│  • Pydantic 스키마로 입력 검증                                │
│  • 인증·인가 의존성 주입 (Depends)                            │
│  • 응답 직렬화 + HTTP 상태 코드 반환                          │
│  • 비즈니스 로직을 포함하지 않음                               │
├─────────────────────────────────────────────────────────────┤
│  Layer 2: Service (비즈니스 로직 레이어)                      │
│  ─────────────────────────────────────────────────────────── │
│  • 핵심 비즈니스 규칙 구현                                    │
│  • 트랜잭션 경계 관리 (async with session)                    │
│  • 다중 Repository 조합                                      │
│  • Celery 비동기 작업 발행                                    │
│  • 외부 서비스 호출 조율 (integrations 경유)                  │
├─────────────────────────────────────────────────────────────┤
│  Layer 3: Repository (데이터 접근 레이어)                     │
│  ─────────────────────────────────────────────────────────── │
│  • SQLAlchemy 쿼리 캡슐화                                    │

### 5.2 미들웨어 체인

HTTP 요청이 라우터에 도달하기 전, 다음 미들웨어가 **순서대로** 실행된다.

```
Client 요청
    │
    ▼
┌──────────────────────────────────────────────────┐
│ ① CORS Middleware                                 │
│   • 허용 Origin 확인, Preflight 응답              │
│   • FastAPI CORSMiddleware (내장)                 │
├──────────────────────────────────────────────────┤
│ ② RequestId Middleware                            │
│   • X-Request-Id 헤더 생성 (UUID v4)              │
│   • structlog에 request_id 바인딩                 │
│   • 응답 헤더에 X-Request-Id 포함                 │
├──────────────────────────────────────────────────┤
│ ③ RateLimit Middleware                            │
│   • slowapi 기반, Redis 카운터                    │
│   • IP + 사용자 기반 이중 제한                    │
│   • 429 Too Many Requests 반환                   │
├──────────────────────────────────────────────────┤
│ ④ Auth Middleware (Depends)                       │
│   • JWT 토큰 검증 (python-jose)                  │
│   • Redis 블랙리스트 확인                         │
│   • current_user 객체 주입                        │

├──────────────────────────────────────────────────┤
│ ⑤ Workspace Middleware (Depends)                  │
│   • X-Workspace-Id 헤더에서 org_id 추출           │
│   • 사용자의 해당 기관 접근 권한 확인              │
│   • PostgreSQL 세션에 app.current_org_id 설정     │
│   │  → RLS 정책 활성화                            │
│   • 403 Forbidden 반환 (권한 없는 기관)           │
├──────────────────────────────────────────────────┤
│ ⑥ RBAC Middleware (Depends)                       │

│   • 403 Forbidden 반환 (역할 부족)               │
└──────────────────────────────────────────────────┘
    │
    ▼
 Router (엔드포인트 핸들러)

### 5.4 의존성 주입

FastAPI의 `Depends()` 체인을 활용하여 계층 간 의존성을 관리한다.

```
엔드포인트 핸들러
    │
    ├── Depends(get_current_user)        ← JWT 검증 → User 객체
    │       │
    │       └── Depends(get_db_session)  ← AsyncSession 생성
    │
    ├── Depends(get_workspace_context)   ← X-Workspace-Id → org_id
    │       │                               + RLS 세션 변수 설정
    │       └── Depends(get_current_user)
    │
    ├── Depends(require_roles(["AM", "AO"]))  ← RBAC 검증
    │       │
    │       └── Depends(get_current_user)
    │
    └── Depends(get_content_service)     ← Service 인스턴스
            │
            ├── Depends(get_db_session)
            └── Depends(get_content_repo)
                    │
                    └── Depends(get_db_session)
```

**핵심 의존성 함수:**

| 함수 | 반환 타입 | 역할 |
|---|---|---|
| `get_db_session` | `AsyncSession` | DB 세션 생성, 요청 종료 시 자동 close |
| `get_current_user` | `User` | JWT → 사용자 조회, 블랙리스트 확인 |
| `get_workspace_context` | `WorkspaceContext` | org_id 추출, RLS 설정, 접근 권한 확인 |
| `require_roles(roles)` | `User` | 역할 목록 대비 현재 사용자 역할 검증 |
| `get_*_service` | `*Service` | 각 도메인 서비스 인스턴스 생성 |

## 6. 멀티테넌트 아키텍처

### 6.1 격리 전략

**논리적 격리(Shared Database, Shared Schema)** 방식을 채택한다.

```
┌──────────────────────────────────────────────┐
│  격리 방식 비교                                │
├──────────────┬─────────────┬─────────────────┤
│              │ 물리 분리    │ 논리 격리 (선택) │
├──────────────┼─────────────┼─────────────────┤
│ DB 인스턴스  │ 기관별 별도  │ 1개 공유        │
│ 스키마       │ 기관별 별도  │ 1개 공유        │
│ 데이터 격리  │ 물리적       │ RLS 정책        │
│ 인프라 비용  │ 높음        │ 낮음            │
│ 기관 추가    │ DB 생성 필요 │ 행 삽입만       │
│ 교차 기관 쿼리│ 불가        │ SA 권한으로 가능│
│ 리스크       │ 낮음        │ RLS 설정 누락 시│
│              │             │ 데이터 유출 가능 │
└──────────────┴─────────────┴─────────────────┘
```

**선택 근거:**
- 위탁기관 수가 수십~수백 수준으로 예상되어, 기관별 DB 인스턴스는 인프라·운영 부담이 과도
- 수탁업체 관리자가 "전체 기관" 뷰로 교차 기관 데이터를 조회해야 하는 요구사항 존재
- PostgreSQL RLS로 행 수준 격리가 가능하며, 인덱스 성능에 미치는 영향이 최소

### 6.2 계층별 격리 메커니즘

모든 계층에서 `organization_id`를 기반으로 데이터를 격리한다.

| 계층 | 격리 방식 | 메커니즘 |
|---|---|---|
| **Database** | PostgreSQL RLS | `organization_id = current_setting('app.current_org_id')` 정책 |
| **API** | X-Workspace-Id 헤더 | 미들웨어에서 org_id 추출 → 세션 변수 설정 |
| **파일 저장소** | org_id 프리픽스 | MinIO 경로: `publisync-media/{org_id}/uploads/...` |
| **캐시** | 키에 org_id 포함 | Redis 키: `cache:dashboard:{org_id}:summary` |
| **비동기 작업** | Celery 작업 인자 | `publish_content.delay(content_id, org_id=org_id)` |
| **SSE** | 채널명에 org_id | Redis Pub/Sub: `sse:channel:{org_id}` |
| **검색** | 테넌트 필터 | Meilisearch: `filter: organization_id = {org_id}` [Phase 2] |

### 6.3 PostgreSQL RLS 설계

#### RLS 정책 정의

```sql
-- 1. 모든 테넌트 격리 대상 테이블에 공통 적용
ALTER TABLE contents ENABLE ROW LEVEL SECURITY;
ALTER TABLE contents FORCE ROW LEVEL SECURITY;

-- 2. 테넌트 격리 정책
CREATE POLICY tenant_isolation ON contents
  USING (organization_id = current_setting('app.current_org_id')::uuid);

-- 3. 시스템 관리자(SA) 우회 정책
CREATE POLICY sa_bypass ON contents
  USING (current_setting('app.user_role', true) = 'SYSTEM_ADMIN');
```

#### 세션 변수 설정 (SQLAlchemy 연동)

```
요청 수신
    │
    ▼
Workspace Middleware (Depends)
    │
    ├── X-Workspace-Id 헤더에서 org_id 추출
    ├── 사용자의 해당 기관 소속 여부 확인 (user.organizations)
    │
    ▼
DB 세션에 세션 변수 설정
    │
    ├── SET LOCAL app.current_org_id = '{org_id}'
    ├── SET LOCAL app.user_role = '{user.role}'
    │
    ▼
이후 모든 SELECT/INSERT/UPDATE/DELETE에 RLS 자동 적용
```

**SQLAlchemy 이벤트 훅:**

```
AsyncSession 생성 시 after_begin 이벤트에서:
├── SET LOCAL app.current_org_id = :org_id
├── SET LOCAL app.user_role = :role
└── 트랜잭션 종료 시 자동 리셋 (SET LOCAL은 트랜잭션 스코프)
```

#### RLS 적용 대상 테이블

| 테이블 | organization_id | 비고 |
|---|---|---|
| contents | FK → organizations | 콘텐츠 |
| comments | FK → organizations | 댓글 |
| channels | FK → organizations | 연동 채널 |
| approvals | FK → organizations | 승인 |
| notifications | FK → organizations | 알림 |
| audit_logs | FK → organizations | 감사 로그 |
| media_assets | FK → organizations | 미디어 [Phase 2] |
| reports | FK → organizations | 리포트 [Phase 3] |
| calendar_events | FK → organizations | 캘린더 [Phase 2] |

**RLS 미적용 테이블 (전역 데이터):**

| 테이블 | 근거 |
|---|---|
| users | 사용자는 여러 기관에 소속 가능 (다대다) |
| organizations | 기관 목록은 전역 조회 필요 |
| roles | 역할 정의는 시스템 전역 |
| system_announcements | 시스템 공지 |

### 6.5 교차 테넌트 접근 방지

| 방어 계층 | 메커니즘 | 우회 조건 |
|---|---|---|
| **API 레이어** | X-Workspace-Id를 사용자의 소속 기관 목록과 대조 | 없음 (미소속 기관 ID는 즉시 403) |
| **DB 레이어** | RLS 정책이 모든 쿼리에 organization_id 필터 자동 추가 | SA 역할만 우회 가능 |
| **파일 레이어** | Presigned URL 생성 시 org_id 프리픽스 경로 제한 | 없음 |
| **캐시 레이어** | Redis 키에 org_id 포함, 키 패턴 제한 | 없음 |
| **테스트** | 교차 테넌트 접근 시도 자동화 테스트 포함 | — |

> **중요:** 새로운 테이블 추가 시 반드시 `organization_id` 컬럼과 RLS 정책을 함께 적용한다. 코드 리뷰 체크리스트에 "RLS 정책 적용 여부" 항목을 포함한다.

### 7.2 JWT 토큰 구조

**Access Token Payload:**

| 필드 | 타입 | 설명 |
|---|---|---|
| `sub` | string | 사용자 ID (`usr_abc123`) |
| `jti` | string | 토큰 고유 ID (블랙리스트용) |
| `role` | string | 사용자 역할 (`SYSTEM_ADMIN`, `AGENCY_MANAGER`, `AGENCY_OPERATOR`, `CLIENT_DIRECTOR`) |
| `org_ids` | string[] | 소속 기관 ID 목록 |
| `iat` | number | 발급 시각 (Unix timestamp) |
| `exp` | number | 만료 시각 (iat + 1800초 = 30분) |

**토큰 설정:**

| 항목 | 값 | 근거 |
|---|---|---|
| 서명 알고리즘 | HS256 | 단일 서비스 구조에서 대칭키로 충분 |
| Access Token TTL | 30분 | 보안(짧은 유효기간)과 UX(빈번한 갱신 방지) 균형 |
| Refresh Token TTL | 7일 | `rememberMe` 옵션 시 30일로 연장 |
| 블랙리스트 TTL | Access Token TTL과 동일 | 만료된 토큰은 자연 무효화 |

### 9.1 큐 토폴로지

3개의 Celery 큐를 분리하여 작업 유형별 독립성을 보장한다.

```
┌─────────────────────────────────────────────────────────────────┐
│  Celery 큐 토폴로지                                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  API Server ──── Redis (Broker) ────── Celery Workers          │
│       │                │                     │                  │
│       │         ┌──────┴──────┐              │                  │
│       │         │             │              │                  │

│       │    │ publish │  │   ai    │  │  system   │           │
│       │    │   큐    │  │   큐    │  │    큐     │           │
│       │    ├─────────┤  ├─────────┤  ├───────────┤           │
│       │    │ 콘텐츠  │  │ 메타데이│  │ 댓글 수집 │           │

│       │    │         │  │ 자막생성│  │ 데이터    │           │

**큐 분할 근거:**

| 큐 | 용도 | 분리 이유 |
|---|---|---|
| `publish` | 콘텐츠 게시, 재시도 | 게시 작업은 시간에 민감(예약 게시)하므로, AI 작업의 지연에 영향받지 않아야 함 |
| `ai` | AI 생성, 감성 분석, 자막, 리포트 | AI API 호출은 응답 시간이 길고(10~60초) 비용이 발생하므로 독립 관리 |
| `system` | 댓글 수집, 토큰 갱신, 알림, 동기화 | 주기적 배치 작업으로 다른 큐에 영향을 주지 않아야 함 |

### 9.3 재시도 전략

작업 유형별 재시도 정책을 테이블로 정의한다.

| 작업 유형 | 최대 재시도 | 백오프 전략 | 타임아웃 | 실패 시 조치 |
|---|:---:|---|:---:|---|
| 콘텐츠 게시 | 3회 | 지수 (5m → 15m → 30m) | 120초 | 알림 발송, 수동 재시도 안내 |
| AI 메타데이터 | 2회 | 고정 (10초) | 30초 | Fallback 메시지 반환 |
| 감성 분석 | 2회 | 고정 (5초) | 20초 | 라벨 미부여 (수동 분류 대기) |
| 댓글 수집 | 3회 | 지수 (1m → 5m → 15m) | 60초 | 다음 주기에 재시도 |
| 토큰 갱신 | 3회 | 지수 (5m → 30m → 2h) | 30초 | 상태 EXPIRED, 알림 발송 |
| 알림 발송 | 2회 | 고정 (30초) | 15초 | 발송 실패 로그 기록 |
| 리포트 생성 | 1회 | — | 300초 | 실패 알림, 재생성 안내 |
| 영상 처리 | 1회 | — | 600초 | 실패 알림, 수동 처리 안내 |

### 9.4 작업 상태 관리

```
작업 상태 머신:

         task.delay()
              │
              ▼
        ┌──────────┐
        │ PENDING  │  큐에 대기 중
        └────┬─────┘
             │ 워커가 작업 수신
             ▼
        ┌──────────┐
        │ STARTED  │  실행 중
        └────┬─────┘
             │
      ┌──────┴──────┐
      │             │
      ▼             ▼
┌──────────┐  ┌──────────┐
│ SUCCESS  │  │ FAILURE  │
│          │  │          │
│ 결과 저장│  │ retry?   │

│          │  │  no  → 알림 발송
└──────────┘  └──────────┘
```

**프론트엔드 진행 상태 표시:**

```
비동기 작업 추적 패턴:
1. API 호출 → 202 Accepted + { jobId: "job_xxx" } 반환
2. 프론트엔드: GET /ai/jobs/:jobId 폴링 (3초 간격)
   또는 SSE 이벤트 수신 (job.completed / job.failed)
3. 진행 상태 표시: 스피너 + 진행률(%) + 예상 잔여 시간
4. 완료 시: 결과 표시 + TanStack Query 캐시 무효화
```

## 10. 실시간 통신 아키텍처

### 10.1 알림 채널 계층

알림의 긴급도와 도달 보장 수준에 따라 4개 채널을 계층적으로 운영한다.

```
┌─────────────────────────────────────────────────────────────────┐
│  알림 채널 계층                                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─── Layer 1: SSE (Server-Sent Events) ─────────────────────┐ │
│  │  • 웹 앱 접속 중인 사용자에게 실시간 전달                    │ │
│  │  • 지연: < 1초                                              │ │
│  │  • 도달 보장: 접속 중인 경우만                               │ │
│  │  • 용도: 뱃지 갱신, 승인 알림, 게시 완료, 댓글 감지         │ │
│  │  • Phase: 1-A                                               │ │
│  └──────────────────────────────────────────────────────────────┘ │
│                                                                 │
│  ┌─── Layer 2: Web Push ─────────────────────────────────────┐  │
│  │  • 브라우저 닫혀 있어도 OS 알림으로 전달                    │  │
│  │  • 지연: 1~5초                                              │  │
│  │  • 도달 보장: Service Worker 등록된 경우                     │  │
│  │  • 용도: 위험 댓글, 승인 요청, 게시 실패                    │  │
│  │  • Phase: 1-B                                               │  │
│  └──────────────────────────────────────────────────────────────┘ │
│                                                                 │
│  ┌─── Layer 3: 텔레그램 Bot ─────────────────────────────────┐  │
│  │  • 외부 메신저로 즉시 전달                                  │  │
│  │  • 지연: 1~3초                                              │  │
│  │  • 도달 보장: 텔레그램 앱 설치 + Bot 연동 시                │  │
│  │  • 용도: 긴급 알림 (위험 댓글, 여론 급변, 시스템 장애)      │  │
│  │  • Phase: 1-B                                               │  │
│  └──────────────────────────────────────────────────────────────┘ │
│                                                                 │
│  ┌─── Layer 4: 이메일 ───────────────────────────────────────┐  │
│  │  • 비실시간, 기록 보관 목적                                 │  │

### 11.3 Human-in-the-Loop 패턴

#### 공통 AI 응답 구조

모든 AI 기능의 응답은 다음 공통 구조를 따른다.

```
{
  "isAiGenerated": true,          // AI 생성 여부 표시
  "confidence": 0.85,             // AI 신뢰도 (0.0 ~ 1.0)
  "fallbackAvailable": true,      // 수동 대체 경로 존재 여부
  "model": "gpt-4o-mini",         // 사용된 모델 (내부 기록용)
  "suggestions": [                // 제안 목록 (복수 후보)
    {
      "content": "제안 내용...",
      "score": 0.85
    }
  ],
  "usage": {                      // 비용 추적
    "promptTokens": 150,
    "completionTokens": 50,
    "totalTokens": 200,
    "estimatedCost": 0.0003       // USD
  }
}
```

#### 피드백 루프

```
사용자 AI 결과 수신
    │
    ├── [적용] → accepted (채택)
    │   └── DB 기록: feedback = "accepted", original + final 저장
    │
    ├── [수정] → modified (수정 후 사용)
    │   └── DB 기록: feedback = "modified", original + edited 저장
    │       → 수정 내용은 프롬프트 개선의 참고 데이터
    │
    └── [무시] → rejected (거부)
        └── DB 기록: feedback = "rejected", reason (선택)
            → 거부 비율이 높으면 프롬프트 재검토 트리거

피드백 데이터 활용:
├── 기관별 채택률 분석 → 기관별 프롬프트 튜닝
├── 기능별 거부 패턴 분석 → 프롬프트 개선
└── 주간 AI 품질 리포트 생성 (시스템 관리자용)
```

### 11.4 Fallback 매트릭스

| 기능 | 타임아웃 시 | AI 장애 시 | 품질 미달 시 |
|---|---|---|---|
| 제목/설명/해시태그 (F02) | "직접 입력하세요" 안내 | 동일 | 사용자가 수정 후 사용 |
| 감성 분석 (F04) | 라벨 미부여, 수동 분류 대기 | 동일 | "미분류" 표시, 수동 태깅 |
| 답글 초안 (F05) | "직접 작성하세요" + 템플릿 제안 | 동일 | 사용자가 수정 후 게시 |
| 톤앤매너 변환 (F17) | 원문 그대로 유지 | 동일 | 사용자가 수동 수정 |
| 표현 검수 (F21) | "AI 검수 불가, 수동 검수" 안내 | 동일 | 검수자 수동 확인 필수 |
| 자막 생성 (F03) | "자막 생성 실패" + 수동 입력 UI | 동일 | 자막 편집기에서 수정 |
| 리포트 생성 (F19) | "리포트 생성 실패, 재시도" 안내 | 데이터만 출력 (AI 분석 없이) | 사용자가 텍스트 수정 |

### 14.2 암호화 계층

| 구분 | 대상 | 방식 | 키 관리 |
|---|---|---|---|
| **전송 암호화** | 모든 외부 통신 | TLS 1.3 (Nginx) | Let's Encrypt 자동 갱신 |
| **전송 암호화** | Redis 연결 | TLS + requirepass | 환경 변수 |
| **저장 암호화 (디스크)** | PostgreSQL 볼륨 | LUKS/dm-crypt | OS 레벨 키 |
| **저장 암호화 (오브젝트)** | MinIO 파일 | SSE-S3 (AES-256) | MinIO KES |
| **필드 암호화** | OAuth refresh_token | Fernet (AES-128-CBC) | 환경 변수 (`OAUTH_ENCRYPTION_KEY`) |
| **필드 암호화** | OAuth access_token | Fernet (AES-128-CBC) | 동일 키 |
| **해싱** | 사용자 비밀번호 | bcrypt (12 라운드) | 솔트 자동 생성 |

### 14.6 CSAP 클라우드 보안 인증 대응

> 범위정의서 §8 "클라우드 보안 인증(CSAP): 공공기관 클라우드 이용 시 CSAP 인증 환경 요구 가능성" 대응

공공기관이 클라우드 환경에서 PubliSync를 운영할 경우, CSAP(Cloud Security Assurance Program) 인증 환경이 요구될 수 있다.

```
CSAP 대응 전략:
├── Phase 1 (현재): 온프레미스 Docker Compose 배포
│   ├── CSAP 인증은 클라우드 이용 시 적용 → 온프레미스에서는 해당 없음
│   └── 단, CSAP 요구사항과 호환 가능한 아키텍처로 설계
│
├── CSAP 호환 아키텍처 요소 (이미 적용):
│   ├── 데이터 암호화: TLS 1.3 + Fernet + LUKS (§14.2)
│   ├── 접근 제어: RBAC + RLS 다중 계층 (§7, §6)
│   ├── 감사 로그: INSERT-ONLY, 3년 보관 (§14.4)
│   ├── 개인정보 보호: 마스킹, 필터링 (§14.3)
│   ├── 네트워크 분리: Docker 내부 네트워크, 외부 노출 최소 (§15.2)
│   └── 취약점 관리: Dependabot + pip-audit (§14.5)
│
├── 클라우드 전환 시 추가 조치 (Phase 2+):
│   ├── CSAP 인증 IaaS/PaaS 환경 선정 (NHN Cloud, KT Cloud 등)
│   ├── 관리적 보안: 보안 정책 문서화, 보안 교육 이력 관리
│   ├── 물리적 보안: CSAP 인증 데이터센터에 의존
│   └── 기술적 보안: WAF, IDS/IPS 적용, 보안관제 연동
│
└── 참고: CSAP 인증은 서비스 제공자(클라우드)의 인증이며,
    PubliSync 자체가 인증 대상은 아님. 다만 CSAP 인증 환경에서
    운영 가능한 아키텍처 호환성을 확보하는 것이 목표.
```

### 18.3 용어 정의

| 용어 | 정의 |
|---|---|
| **RLS** | Row-Level Security. PostgreSQL의 행 수준 보안 정책으로, 쿼리 실행 시 자동으로 행 필터링 |
| **SSE** | Server-Sent Events. HTTP 기반 서버→클라이언트 단방향 실시간 통신 프로토콜 |
| **Fernet** | 대칭 암호화 방식 (AES-128-CBC + HMAC-SHA256). cryptography 라이브러리에서 제공 |
| **VAPID** | Voluntary Application Server Identification. Web Push 알림 전송 시 서버 신원 증명 키 |
| **Presigned URL** | 제한된 시간 동안 특정 S3 객체에 접근할 수 있는 서명된 URL |
| **litellm** | 100+ LLM 제공자를 통일된 인터페이스로 추상화하는 Python 라이브러리 |
| **Celery Beat** | Celery의 주기적 작업 스케줄러. crontab 또는 interval 기반으로 작업 자동 발행 |
| **Flower** | Celery 작업 큐의 실시간 모니터링 웹 대시보드 |
| **백프레셔** | Backpressure. 시스템 과부하 시 요청 유입 속도를 자동으로 조절하는 메커니즘 |
| **PITR** | Point-in-Time Recovery. WAL 아카이빙 기반으로 특정 시점으로 DB를 복원하는 기술 |
| **C4 Model** | Context, Container, Component, Code 4단계로 소프트웨어 아키텍처를 시각화하는 모델링 방법론 |
